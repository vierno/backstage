Types and validators that help describe the model of a Backstage Catalog

Classes

ClassDescription
CommonValidatorFunctionsContains various helper validation and normalization functions that can be composed to form a Validator.
DefaultNamespaceEntityPolicySets a default namespace if none was set.
FieldFormatEntityPolicyEnsures that the format of individual fields of the entity envelope is valid.This does not take into account machine generated fields such as uid, etag and generation.
KubernetesValidatorFunctionsContains validation functions that match the Kubernetes spec, usable to build a catalog that is compatible with those rule sets.
NoForeignRootFieldsEntityPolicyEnsures that there are no foreign root fields in the entity.
SchemaValidEntityPolicyEnsures that the entity spec is valid according to a schema.This should be the first policy in the list, to ensure that other downstream policies can work with a structure that is at least valid in therms of the typescript type.

Functions

FunctionDescription
compareEntityToRef(entity, ref, context)Compares an entity to either a string reference or a compound reference.The comparison is case insensitive, and all of kind, namespace, and name must match (after applying the optional context to the ref).
entityEnvelopeSchemaValidator(schema)Creates a validation function that takes some arbitrary data, and either returns that data cast to an EntityEnvelope (or the given subtype) if it matches that schema, or throws a describing the errors.Note that this validator is only meant for applying the base schema checks; it does not take custom policies or additional processor based validation into account.By default, the plain EntityEnvelope schema is used. If you pass in your own, it may contain $ref references to the following, which are resolved automatically for you:- EntityEnvelope - Entity - EntityMeta - common#
entityHasChanges(previous, next)Checks whether there are any significant changes going from the previous to the next version of this entity.Significance, in this case, means that we do not compare generated fields such as uid, etag and generation.Note that this comparison does NOT take status, relations or similar into account. It only compares the actual input entity data, i.e. metadata and spec.
entityKindSchemaValidator(schema)Creates a validation function that takes some arbitrary data, and either returns that data cast to a T if it matches that schema, or false if the schema apiVersion/kind didn't apply to that data, or throws a describing actual errors.This validator is highly specialized, in that it has special treatment of the kind and apiVersion root keys. This only works if your schema has their rule set to "enum":
 
"apiVersion": {
   "enum": ["backstage.io/v1alpha1", "backstage.io/v1beta1"]
},
"kind": {
  "enum": ["Group"]
},

Type Aliases

Type AliasDescription
EntityThe parts of the format that's common to all versions/kinds of entity.
EntityEnvelopeThe envelope skeleton parts of an entity - whatever is necessary to be able to give it a ref and pass to further validation / policy checking.
EntityLinkA link to external information that is related to the entity.
EntityMetaMetadata fields common to all versions/kinds of entity.
EntityNameA complete entity name, with the full kind-namespace-name triplet.
EntityPolicyA policy for validation or mutation to be applied to entities as they are entering the system.
EntityRefA reference by name to an entity, either as a compact string representation, or as a compound reference structure.The string representation is on the form [:][/].Left-out parts of the reference need to be handled by the application, either by rejecting the reference or by falling back to default values.
EntityRefContextThe context of defaults that entity reference parsing happens within.
